# Design: `/prove-missing` Workflow

**Date:** 2026-03-03
**Scope:** General workflow for formalizing theorems whose proofs are missing from the Bubeck convex optimization book (and similar textbooks).

---

## Problem

Several theorems in Bubeck's book are cited without proof (e.g., Lemma 2.2 — Grünbaum's theorem, cited as [Gru60]). The existing `/prove-theorem` skill assumes an informal proof is available in the section file. When no proof is given, `lean4:autoprove` starts from a bare statement with little guidance, leading to poor results.

## Goal

A repeatable, Claude-driven workflow that:
1. Generates a structured informal proof from web sources and Claude's own knowledge
2. Uses that proof as a bridge to drive Lean 4 formalization
3. Handles failure gracefully — never blocks on a hard theorem
4. Produces best-effort output (sorrys allowed for genuine Mathlib gaps)

---

## Architecture

New skill: `/prove-missing "<ID>"` — wraps existing tools, inserting an informal proof generation step before formalization.

```
/prove-missing "<ID>"
       │
       ▼
1. Read section file → extract theorem statement + hypotheses
       │
       ▼
2. WebSearch (3–5 queries) → gather informal proofs from literature
       │
       ▼
3. Synthesize structured informal proof → save to
   proofs/<book>/informal/<TheoremName>_informal.md
       │
       ▼
4. lean4:autoprove (uses informal proof as additional context)
       │
       ▼
5. Update section YAML status + append to logs/proofs.log
```

The informal proof file persists on disk — formalization can be re-run without re-querying the web.

---

## Informal Proof Format

Saved to `proofs/<book>/informal/<TheoremName>_informal.md`:

```markdown
# Informal Proof: <Theorem Name>

## Sources
- [Source 1](url)
- [Source 2](url)

## Theorem Statement
<Full statement in plain math>

## Proof Steps

### Step N: <Title>

**English:** Full sentence explanation of what this step does and why.

**Math:** The key equation/inequality being established.
  e.g., Vol(𝒦_t) ≥ Vol(𝒦_{t'}) · (t/t')^(n-1)

**Lean/Mathlib hint:** candidate lemma names or search terms

**Sorry prediction:** [yes/no] — flag if this step likely needs `sorry`

## Mathlib Gaps
- <Gap description>: not found in Mathlib 4.x

## Status
[PROOF FOUND | CLAUDE-GENERATED (unverified) | PROOF NOT FOUND — manual input needed]
```

The full **English + Math** per step is the key improvement over a bare step skeleton — it gives `lean4:autoprove` the logical flow needed to choose the right tactics.

## WebSearch Strategy

Run 3–5 targeted queries in sequence:

| # | Query template | Purpose |
|---|---|---|
| 1 | `"<theorem name> proof"` | Find the classical proof directly |
| 2 | `"<theorem name> convex geometry proof sketch"` | Broaden if query 1 is sparse |
| 3 | `"<theorem name> Lean 4 Mathlib formalization"` | Check if already formalized |
| 4 | `"<key ingredient> Mathlib"` | Verify supporting lemmas exist |
| 5 | `"<theorem name> lecture notes"` | Fallback for pedagogical write-ups |

**Source priority** (highest → lowest):
1. Existing Lean/Mathlib formalizations — if found, use directly
2. Wikipedia — reliable for classical results
3. arXiv lecture notes / survey papers
4. Math StackExchange

**Synthesis rule:** when multiple sources agree, use the most formalization-friendly version (fewest implicit steps). When sources conflict, note it and use the majority version.

**Fallback chain:**
```
WebSearch finds proof → use it
     ↓ (no useful results)
Claude generates from own knowledge → flag as "Claude-generated, unverified"
     ↓ (Claude uncertain)
Document the gap, mark theorem sorry, move on
```

## Integration with Existing Workflow

| Skill | Role | Changed? |
|---|---|---|
| `prove-theorem` | Full pipeline for theorems with proofs in the book | No |
| `lean4:autoprove` | Lean formalization engine | No |
| `lean4:checkpoint` | Commit verified result | No |
| `prove-missing` *(new)* | Wraps above + adds WebSearch + informal proof | New skill file |

**File layout:**
```
proofs/<book>/
  informal/
    <TheoremName>_informal.md    ← generated by /prove-missing, persists
  <TheoremName>.lean             ← filled by autoprove
```

**How autoprove uses the informal proof:**
Claude reads `_informal.md` and works through each step sequentially:
- Translates **Math** → Lean term
- Uses **Lean/Mathlib hint** to seed lemma searches
- Places `sorry` where **Sorry prediction** flags a gap

---

## Failure Handling

**1. Informal proof generation fails**
- Save minimal `_informal.md` with `## Status: PROOF NOT FOUND`
- Skip autoprove, mark `status: partial`, log `status: "failed"`
- Do not block — move to next theorem

**2. Formalization gets stuck**
- Place `sorry` with comment: `-- TODO: needs <lemma name>`
- Continue with remaining steps
- Mark `status: partial`

**3. Required Mathlib lemma doesn't exist**
- Document in `## Mathlib Gaps` section of `_informal.md`
- Lean file `sorry` gets a structured comment pointing to the gap

**Overall failure flow:**
```
prove-missing "<ID>"
  ├── informal proof: OK → autoprove → proved/partial
  ├── informal proof: OK → autoprove → stuck → partial + sorry + gap note
  ├── informal proof: fallback to Claude knowledge → flagged → autoprove
  └── informal proof: not found → skip autoprove → failed + documented
```

---

## Success Criteria

- `/prove-missing "Lemma 2.2"` runs end-to-end without manual intervention
- `_informal.md` exists with full English + math proof and Mathlib hints
- `Lemma22.lean` has maximum proof coverage (sorrys only for genuine Mathlib gaps)
- `logs/proofs.log` has a new entry
- `index.md` status updated

---

## Out of Scope

- OpenAI API integration (requires separate API account; revisit if access obtained)
- Automated theorem discovery (still manual — user specifies theorem ID)
- Proving theorems that require entirely new Mathlib infrastructure
